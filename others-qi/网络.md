# http和https的区别
## 概念
http:超文本传输协议。是目前发布和接收html页面使用非常广泛的一种网络协议。它的信息采用明文传输。可以被直接抓取，故安全性不强。不验证通信方身份，可能遭到伪装；无法验证报文的完整性，可能被篡改。
https：http的加密版协议，内部封装了ssl加密通道，以安全为目标。它的安全基础是ssl。http是基于http标准并对tcp传输数据进行加密的协议。
## 区别
1. https协议需要CA申请证书.
2. http是超文本传输协议，信息是明文传输。https则是具有安全性的ssl加密传输协议。
3. http和https使用的是完全不同的连接方式，用的端口也不一样。前者是80，后者是443
4. http连接很简单，是无状态（数据包的发送、传输和接收都是相互独立的）的。https协议是由ssl+http协议固件的可进行加密传输、身份认证的网络协议，比http协议安全。
## ssl
一种加密通道的规范，是https的安全基础。ssl协议位于http之下，tcp之上的一个协议层。由网景公司开发。
###
1. 对称加密：单钥密码系统的加密方式，即用同一个秘钥来加密和解密信息。
2. 公钥私钥：a有公钥和私钥，它把公钥发给b,b把要传送的内容用公钥加密之后发给a，a再用私钥去解密拿到信件内容。

## 常用的http请求方法有哪些
1. get：向特定路径发出请求
2. post：向指定路径资源提交数据进行处理请求
3. put
4. head:向服务器索与get请求相一致的响应，但响应体不会被返回
5. delete：请求服务器删除指定页面
6. options返回服务器针对特定资源所支持的http请求方法

## get和post方法的区别（基于http协议的两种最常用的请求方式）
1. 侧重点不同：get请求主要用来向服务端获取信息，post请求主要用来向服务端提交信息。比如从服务器获取一个页面，获取一个图片或者一些简单的展示信息可以用get请求 。当需要向服务端提交大量的信息，比如客户端编辑了一篇文章，需要把这篇文章提交到服务端，那最好的方式就是post请求。
2. 请求参数位置不同：get请求是把请求参数和对应的值直接拼接在url链接后（通过？连接，多个请求数据间用&连接）如http://127.0.0.1/Test/login.action?name=admin&password=admin。而post请求则是将请求参数封存在请求体中发送给服务器，这个过程对用户是不可见的，
3. 传输数据大小：由于受url的限制（2048字节），get传输的数据不易过多，但效率高。而post对于数据大小无限制。
4. 安全性：post较get安全性更高。因为get相当于将其参数直接暴露在url上，所以可能会泄露私密信息。
5. 支持字符：get方式只能支持ascii字符，向服务器穿的中文字符可能会乱码。post支持标准字符集，可正确传递中文字符。
6. post请求需要一次预检过程，也就是会先发送option请求，待预检通过才会发送数据包，但get请求只用发送一次请求。所以在某种情况下，post请求比get请求更耗费时间。
# http请求和响应报文格式
1. 请求报文：请求行（请求方式、url、http版本号）；请求头header(键值对形式，比如客户端可接收的内容类型编码方式等);请求体body（post方法中，会把数据以键值对形式发送）；
2. 响应：状态行（http版本号+状态码+状态码文本描述）；响应头（包括服务器类型日期长度）；响应体（服务器返回的html页面或json数据）

# tcp和udp
是传输层的两种协议。tcp(可靠)是一种的面向连接的协议，保证数据的可靠性。即在收发数据之前，必须先确保和对方建立了一个可靠的连接。具体来说，它是先采用三次握手建立一个连接，然后发送需要传输的数据，待数据传输完毕后再通过四次挥手断开一个连接。udp(速度快)是无连接的，传输数据之前并不进行确认，不理会对方能不能收到数据会不会按顺序收到数据，可能产生丢包现象，即不保证可靠交付。常见的视频播放卡带就是由于udp传输协议引起。
+ 三次握手：是为了确保双方都能收到对方的数据包，以此来建立连接。具体来讲，首先，第一次握手，客户端(client)向服务端(server)发送一个数据包，其中同步序列号为j，等待服务器确认；然后第二次握手，服务端将收到的序列号加一作为确认号，同时也发送一个自己的数据包同步序列号为k；第三次握手，客户端收到了服务端发来的确认号数据确认服务端能够收到自己的数据包，同时它又将收到的服务端发来的数据包里的序列号k加一之后作为确认号再次返回给服务端。此包发送完毕，客户端服务端都能收到对方的数据包，也知道自己的数据包能被对方收到，第三次握手完成，tcp连接完毕。
+ 四次挥手：数据传输完毕后，首先客户端向服务端提出停止连接的请求，服务端收到后对其作出响应并将其传递回客户端，告诉客户端我已经知道了。然后，服务端再向客户端提出反方向的断开连接请求，客户端收到请求后确认请求，并告诉服务端，我也知道了。至此，四次四次握手完毕，双方断开连接。

# 常见http状态码（response code）
+ 总：1xx-表示临时响应；2xx--表示请求成功；3xx--表示重定向（要完成请求必须有更进一步的操作）；4xx--b表示请求错误；5xx--表示服务器端错误。
+ 200-成功
+ 204-请求被受理，但无资源可以返回
+ 301-永久性重定向
+ 302-临时重定向
+ 400-请求报文语法有误，服务器无法识别
+ 401-请求需要认证
+ 403-请求资源禁止访问
+ 404-请求失败，服务器找不到请求资源
+ 500-服务器内部错误
+ 502-错误网关
+ 503-服务器不可用
+ 504-网关超时

# 进程和线程
我觉得可以把进程和线程它类比为一个程序和一个任务在动态运行的过程。进程的概念要大于线程，或者说线程是是进程的一部分。具体来讲，进程是操作系统资源分配的的基本单位，而线程是任务调度和执行的基本单位。在同一个进程中可以有多个线程同时执行，每个进程会分配有不同的内存空间，而线程组之间是可以共享资源的。
# 内存溢出
内存不足，系统没有多余的能力去分配程序所需要的空间
# 内存泄漏
程序中已经分配的内存用完后未能释放或者无法释放，造成了系统内存的的浪费。

# cookie和session
它们都是指用来辨别用户身份的数据。但二者有很多的不同点。
+ 首先最显著的区别就是存在的位置不同。cookie存放于客户端，而session存在于服务端。cookie像是“客户身上携带的通行证”，它会跟随请求一同提交至服务器，以此辨别用户信息。而session类似于服务端保存的“客户明细表”，当客户端提交请求时，服务端检索内部session从而确认用户信息。
+ 安全性：session比cookie更安全。cookie是以明文的方式存放在客户端的，安全性低，
一旦被人截获就可能伪装身份冒充登录，而session数据存放于服务器内存中，是无法伪造的。
+ 内容：cookie保存的是字符串，session是对象
+ 访问范围：一个session为一个用户浏览器独享。当访问用户增多，session会比较占用服务器内存，影响性能。
+ session需要借助cookie才能正常；session不能区分路径，同一个用户在访问一个网站期间，所有的session在任何一个地方都可以访问到。而cookie中如果设置了路径参数，那么同一个网站中不同路径下的cookie互相是访问不到的

## cookie属性;

## cookie被禁止，session能正常使用吗
由于cookie可以被人为的禁止，必须有其它的机制以便在cookie被禁止时仍然能够session id传递回服务器，经常采用的一种技术叫做URL重写，就是把session id附加在URL路径的后面
